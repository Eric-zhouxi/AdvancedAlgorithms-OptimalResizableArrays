## 摘要 (Abstract)

> 可调整数组（resizable array）是一种可以从末端（或两端）添加或删除元素、同时支持通过索引在常数时间内访问每个元素的数组。由于数组的大小（即其中元素的数量）随时间变化，空间高效地维护可调整数组需要动态内存管理。标准的倍增技术允许使用仅 O(N) 空间维护大小为 N 的数组，每次操作 O(1) 均摊时间，甚至 O(1) 最坏情况时间。Sitarski 和 Brodnik 等人描述了更好的解决方案，使用仅 N + O(√N) 空间维护大小为 N 的可调整数组，每次操作仍为 O(1) 时间。Brodnik 等人给出了一个简单的证明，表明这在某种意义下是最优的。

> 我们区分了**存储**可调整数组和访问其元素所需的空间，与增长或收缩数组时可能**临时**需要的空间。对每个整数 r ≥ 2，我们证明：如果增长和收缩操作期间可以短暂使用 N + O(N^(1−1/r)) 空间，则 N + O(N^(1/r)) 空间足以存储和访问大小为 N 的数组。通过索引访问元素在最坏情况下只需 O(1) 时间，而增长和收缩操作需要 O(r) 均摊时间。通过对一个**增长游戏**（growth game）的精确分析，我们证明：对于来自一大类数据结构的任何仅使用 N + O(N^(1/r)) 空间存储数组的数据结构，增长操作的均摊代价是 Ω(r)，即使只允许增长和访问操作。除非 r = 2，增长和收缩操作的时间不能达到最坏情况。

## 1. 引言 (Introduction)

数组和可调整数组可能是使用最广泛的数据结构。令人惊讶的是，我们发现关于它们似乎并非所有事情都已经被说尽了。我们描述了可调整数组的新的简单实现，在许多情况下比迄今为止提出的所有实现都更节省内存。我们的视角是理论性的，但我们提出的实现可能具有一定的实际意义。

### 基本概念

- **大小为 N 的数组**：存储序列 a₀, a₁, ..., a_{N-1}，第 i 个元素可在常数时间内被检索或修改。
- **可调整数组**（resizable array / dynamic array / dynamic table）：大小可以增减的数组，同时保持常数时间的随机访问。
- **减小数组**：简单，只需不使用已分配的部分内存（但不节省空间）。
- **增大数组**：更难，因为相邻的内存可能已被其他用途占用。

### 标准方案：倍增技术

标准教科书方案（如 Cormen et al. [4]）：
- 初始分配某个大小的数组
- 数组满时，分配一个更大的（如两倍大小）数组，复制所有元素
- 数组太空时（如使用率 < 1/4），分配一个更小的（如一半大小）数组

缺点：最多可能有 50% 的已分配内存被浪费。如果使用增长因子 1+α，则浪费比例为 1/(1+α)。选择较小的 α 可以减少浪费，但均摊代价变为 O((1+α)/α)。

### 突破性结果

Sitarski [15] 和 Brodnik et al. [2] 描述了仅使用 **N + O(√N)** 空间的简单实现，同时保持 O(1) 操作时间。Brodnik 等人还证明这在某种意义下是最优的：任何可调整数组的实现有时必须使用 N + Ω(√N) 空间。

### 本文贡献

我们提出区分**存储数组所需空间**与**调整大小时临时需要的空间**：

- **对于 r ≥ 2**：存储只需 N + O(N^(1/r)) 空间，临时调整需要 N + O(N^(1−1/r)) 空间
- 访问：O(1) 最坏情况时间
- 增长/收缩：O(r) 均摊时间
- 通过**增长游戏**的精确分析，证明 O(r) 均摊时间是下界（最优的）

## 2. 可调整数组的形式化定义 (Resizable Arrays)

可调整数组支持以下操作：

| 操作 | 说明 |
|------|------|
| `A ← Array()` | 创建空数组 |
| `A.Length()` | 返回当前长度 |
| `A.Get(i)` | 返回第 i 个元素 (0 ≤ i < Length) |
| `A.Set(i, a)` | 修改第 i 个元素 |
| `A.Grow(a)` | 在末尾添加元素 a，长度 +1 |
| `A.Shrink()` | 删除末尾元素，长度 −1 |

### 定义 2.1 ((s(N), t(N))-实现)

设 s(N), t(N) 是两个非递减函数。一个可调整数组数据结构称为 **(s(N), t(N))-实现**，如果：
- 存储大小为 N 的数组使用不超过 N + s(N) 空间
- 对大小为 N 的数组进行增长或收缩操作期间，使用不超过 N + t(N) 空间

## 3. 前人的工作 (Previous Work)

### 3.1 基本数据结构

最简单的空间高效实现：保持一个恰好等于数组大小的定长数组。每次增长/收缩后需要分配新数组。
- 存储：N + O(1) 空间
- 代价：增长/收缩均摊 Ω(N)，且临时需要 2N + O(1) 空间

实用的几何扩展/收缩技术：
- 选择参数 α > 0
- 分配数组满时，分配为 (1+α)N 的新数组
- 使用率低于 1/(1+α)² 时，分配为 N/(1+α) 的新数组
- 增长均摊代价：(1+α)/α
- 收缩均摊代价：1/α

### 3.2 Sitarski 的 HAT 数据结构

Sitarski [15] 提出了**哈希数组树（HAT）**：
- 维护一个大小为 B 的索引块，其中 √N ≤ B < 4√N
- ⌈N/B⌉ 或 ⌈N/B⌉+1 个大小为 B 的数据块
- 总空间：N + 3B + 2 = N + O(√N)
- 第 i 个元素存储在第 ⌊i/B⌋ 个数据块的第 (i mod B) 个位置
- 当 N = B² 时重建（B 加倍），均摊代价 O(1)

### 3.3 Brodnik 等人的数据结构

Brodnik et al. [2] 描述了两种变体：

**变体 (a)**：使用大小为 1, 2, 3, ..., k 的数据块存储元素。
- 1+2+...+k = k(k+1)/2 ≈ k²/2，需要约 √(2N) 个块
- 索引块大小 B = Θ(√N)
- 增长/收缩：O(1) 最坏情况时间（不在后台复制时）
- 访问元素：需计算 ℓ = ⌈(√(8i+1)−1)/2⌉，涉及平方根运算

**变体 (b)**（避免平方根）：每个数据块大小为 2 的幂。
- 对于 k = 0, 1, ...，有一个虚拟超块（super block），由 2^{⌊k/2⌋} 个大小为 2^{⌈k/2⌉} 的块组成
- 通过移位操作即可定位元素

## 4. 下界及其扩展 (The Lower Bound)

### 定理 4.1 (Brodnik 等人的下界)

任何维护可调整数组的数据结构，在某些时刻必须使用 **N + Ω(√N)** 空间，即使只执行增长和访问操作。

**证明思路**：设 k 为 N 次增长操作后使用的连续内存块数量。
- 如果 k ≥ √N：至少需要 N + k ≥ N + √N 空间（指针占用）
- 如果 k < √N：最大的块大小 ℓ > √N，该块被分配时不含元素，但 ℓ 被计入总空间，所以当时总空间 ≥ N' + ℓ > N' + √N'

### 定理 4.2 (本文的扩展)

任何 (s(N), t(N))-实现必须满足 **s(N) · t(N) ≥ N**，即使只支持增长和访问操作。

**推论 4.3**：对于任意 r ≥ 1，任何使用 N + O(N^(1/r)) 空间存储可调整数组的数据结构，在增长操作期间必须偶尔使用 **N + Ω(N^(1−1/r))** 空间。

## 5. 简单的 (O(N^(1/3)), O(N^(2/3)))-实现

本节描述 r=3 时的一个简单实现。

基本思想：
- 从 HAT 数据结构开始，但数据块大小约为 N^(2/3)
- 索引块长度约为 N^(1/3)
- 由于最后一个数据块可能几乎是空的，总空间为 N + O(N^(2/3))
- **关键改进**：再用一个 HAT 来处理最后一个部分填充的数据块！
- 额外空间变为 O((N^(2/3))^(1/2)) = **O(N^(1/3))**

数据结构维护一个参数 B，N^(1/3) ≤ B ≤ 4N^(1/3)：
- 约 N/B² 个大小为 B² 的**大块**（总是满的）
- 最多 2B 个大小为 B 的**小块**（最多一个部分填充，最多一个为空）
- 两个索引块（一个用于大块，一个用于小块）

操作：
- 增长：通常在小块末尾添加即可
- 当 2B 个小块满时：将它们复制到一个新的大块中
- 当没有非空小块时：将一个大块拆分为 B 个小块
- 所有操作 O(1) 时间（增长和收缩均摊，访问最坏情况）

## 6. 一般情况：(O(rN^(1/r)), O(N^(1−1/r)))-实现

### 核心思想

对于给定的 r ≥ 2，维护一个参数 B，N^(1/r) ≤ B < 4N^(1/r)。

将元素存储在大小分别为 B, B², B³, ..., B^(r−1) 的**块**中。
- 大小为 B², B³, ..., B^(r−1) 的块始终是满的
- 大小为 B 的块，除最后一个外都是满的

这模拟了一个**基数 B 计数器**的操作：
- 有 B 个满的大小为 B^i 的块时 → 合并为一个大小为 B^(i+1) 的块
- 需要拆分时则反向操作

### 空间分析

- 总块数：最多 2B(r−1) + 1
- 额外空间：O(rB) = **O(rN^(1/r))**
- 临时额外空间（分配新的大块时）：**O(N^(1−1/r))**

### 增长操作 (Grow)

```
Grow(a):
  如果 N = B^r: Rebuild(2B)        // 数据满了，B 加倍重建
  否则如果 n₁ = 2B 且 n₀ = B: Combine-Blocks()  // 底层满了，向上合并
  否则: 将 a 放入最后一个大小为 B 的块的空位中
```

### 收缩操作 (Shrink)

类似，删除末尾元素。使用**冗余基数 B 计数器**（允许最多 2B 个块而非 B 个）来保证均摊复杂度。

### 定理 6.1

该实现是 (O(rN^(1/r)), O(N^(1−1/r)))-实现：
- 增长和收缩均摊代价：**O(r)**
- 访问操作：**O(1)** 最坏情况时间

### 推论 6.2

取 r = log N，得到：
- 存储：N + O(log N) 空间
- 增长/收缩：O(log N) 均摊时间
- 访问：O(1) 最坏情况时间

## 7. 数据结构变换 (A Data Structuring Transformation)

如果 r = r(N) 是随 N 增长的函数，能否将额外存储空间从 O(rN^(1/r)) 降低到 **O(N^(1/r))**？

### 引理 7.2

一个标准的 (s(N), O(N))-实现可以转换为 **(s(N) + O(N/t(N)), s(N) + O(t(N)))**-实现，其中 t(N) 是任意非递减函数。

核心技巧：当需要分配大小为 b 的大块时，如果 b > t(N)，改为分配 ⌈b/t(N)⌉ 个大小为 t(N) 的小块。

### 定理 7.3

对于 r(N) ≤ (1/2)(log N / log log N)，存在 **(O(N^(1/r)), O(N^(1−1/r)))**-实现：
- 访问 O(1) 最坏情况时间
- 增长/收缩 O(r) 均摊时间

## 8. 增长游戏 (The Growth Game)

为了证明 O(r) 均摊时间是下界，我们定义并完全分析了一个**增长游戏**。

### 游戏规则

参数 N, k, ℓ：
- 将 N 个元素逐个插入 k 个子数组 A₁, A₂, ..., A_k
- 除第一个非空子数组最多有 ℓ 个空位外，其他子数组都是满的
- 额外空间：始终不超过 k + ℓ

**三种操作**：
1. 有空位 → 直接放入（代价：1）
2. 有空子数组 → 分配大小为 ℓ+1 的新子数组（代价：1）
3. 所有 k 个子数组都满 → 选择 i ∈ [k]，合并 A₁,...,A_i 到新子数组中，放入新元素（代价：1 + Σaⱼ）

### 定理 8.6（精确解）

令 C_{N,k} 为 (N,k)-增长游戏的最小总代价。

当 binom(n+k-1, k) ≤ N < binom(n+k, k) 时：
- **C_{N,k} = (N+1)n − binom(n+k, k+1)**
- 均摊代价 A_{N,k} ≥ (k/(k+1))(n−1) ≥ (1/2)(n−1)

### 推论 8.13

当 N = binom(n+k, k) − 1 时（恰好有一个最优最终状态）：
- **C_{N,k} = (kn/(k+1))(N+1)**

## 9. 增长操作均摊代价的下界 (Lower Bound)

### 定理 9.1

任何标准的可调整数组数据结构，如果仅使用 N + O(rN^(1/r)) 空间存储大小为 N 的数组（其中 r = O(log N)），则增长操作的**均摊代价必须为 Ω(r)**。

**证明思路**：
- 标准实现的操作对应于 (N, rN^(1/r), rN^(1/r))-增长游戏中的移动
- 取 k = rN^(1/r), ℓ = rN^(1/r)，应用增长游戏的分析
- 当 n = Ω(r) 时，均摊代价 = Ω(r)
- 使用二项式系数的上界 binom(n, k) ≤ (en/k)^k 完成证明

## 10. 总结 (Concluding Remarks)

1. 我们给出了可调整数组实现的一个**最优参数化族**，在存储空间、调整空间和均摊代价之间取得平衡
2. 技术上核心贡献：
   - 基于基数 B 计数器的**分层块结构**
   - 精确分析**增长游戏**获得匹配下界
   - 简单的**数据结构变换**降低存储开销
3. 下界目前仅适用于"标准"算法，作者相信可以扩展到所有算法
4. 开放问题：是否有更简单的方法（如归纳法）获得增长游戏的下界？
